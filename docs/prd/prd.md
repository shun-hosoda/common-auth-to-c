# Product Requirement Document (PRD)

## 1. 概要 (Overview)

| 項目 | 内容 |
|------|------|
| プロダクト名 | common-auth-to-c |
| バージョン | 0.1.0 |
| ステータス | Draft |
| 作成日 | 2026-04-16 |
| 更新日 | 2026-04-16 |

### 1.1 背景と課題

toC（一般消費者）向けSaaSサービスにおいて、認証・認可機能は必須の基盤である。
しかし、認証基盤はセキュリティ要件が厳しく、ゼロから構築するコストが高い。
複数のSaaSプロダクトで共通利用できる認証基盤を構築することで、開発効率とセキュリティ品質を両立する。

### 1.2 目的

- toC向けSaaSに共通利用可能な認証・認可基盤を提供する
- メール/パスワード認証とソーシャルログインをMVPとして実装する
- セキュリティベストプラクティスに準拠した安全な認証フローを実現する
- 将来的な認証方式の拡張（MFA、パスキー等）に対応できる設計とする

## 2. ターゲットユーザー

| ペルソナ | 説明 | 主な利用シーン |
|----------|------|----------------|
| エンドユーザー（消費者） | SaaSサービスを利用する一般消費者 | アカウント登録、ログイン、プロフィール管理 |
| サービス開発者 | この認証基盤を組み込むSaaS開発チーム | 認証APIの統合、ユーザー管理 |

## 3. 機能要件 (Functional Requirements)

### 3.1 MVP（最小実行可能プロダクト）

| ID | 機能 | 優先度 | 説明 |
|----|------|--------|------|
| FR-001 | メール/パスワード登録 | Must | メールアドレスとパスワードでアカウント作成 |
| FR-002 | メール/パスワードログイン | Must | 登録済みメール/パスワードで認証 |
| FR-003 | メール確認（メール認証） | Must | 登録時にメールアドレスの所有確認 |
| FR-004 | パスワードリセット | Must | メール経由でパスワードを再設定 |
| FR-005 | ソーシャルログイン（Google） | Must | Googleアカウントでログイン |
| FR-006 | ソーシャルログイン（Apple） | Should | Apple IDでログイン |
| FR-007 | トークン管理 | Must | アクセストークン/リフレッシュトークンの発行・更新・失効 |
| FR-008 | ログアウト | Must | セッション破棄・トークン無効化 |
| FR-009 | プロフィール管理 | Should | ユーザー情報の取得・更新 |
| FR-010 | アカウント削除（退会） | Should | ユーザー自身によるアカウント削除 |

### 3.2 将来的な拡張

| ID | 機能 | 説明 |
|----|------|------|
| FT-001 | 多要素認証（MFA/2FA） | TOTP, SMS等による二段階認証 |
| FT-002 | パスキー（WebAuthn/FIDO2） | パスワードレス認証 |
| FT-003 | マジックリンク認証 | メールリンクによるパスワードレスログイン |
| FT-004 | SMS/電話番号認証 | 電話番号ベースの認証 |
| FT-005 | 追加ソーシャルプロバイダ | LINE, Twitter, Facebook等 |
| FT-006 | マルチテナント対応 | テナントごとの認証設定分離 |
| FT-007 | 管理者機能 | ユーザー管理・監査ログ |

## 4. 非機能要件 (Non-Functional Requirements)

| カテゴリ | 要件 | 目標値 |
|----------|------|--------|
| パフォーマンス | 認証APIレスポンスタイム | < 200ms (p95) |
| パフォーマンス | トークン検証レスポンスタイム | < 50ms (p95) |
| 可用性 | アップタイム | 99.9% |
| セキュリティ | パスワードハッシュ | bcrypt or argon2id |
| セキュリティ | トークン方式 | JWT (短命アクセストークン + リフレッシュトークン) |
| セキュリティ | ブルートフォース対策 | レート制限（ログイン試行5回/分） |
| セキュリティ | 通信暗号化 | TLS 1.2以上必須 |
| スケーラビリティ | 同時接続数 | 10,000+ （水平スケール可能な設計） |
| データ保護 | 個人情報保護法準拠 | 同意管理・削除権の保証 |

## 5. 制約事項

- 技術スタック未定：要件定義承認後、設計工程で決定する
- インフラ未定：設計工程でクラウドプロバイダ・構成を決定する
- 初期はモノリス or モジュラーモノリスで開始し、必要に応じてマイクロサービス化

## 6. 成功指標 (KPIs)

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| ログイン成功率 | > 99% | 認証API成功/総試行 |
| ログイン離脱率 | < 5% | ログインページ離脱/表示 |
| 認証APIレスポンス | p95 < 200ms | APMモニタリング |
| セキュリティインシデント | 0件 | 監査ログ・アラート |
